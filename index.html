Mark — Multi-Agent Marketing Automation AI Agents · 2026

OpenClaw · Docker · OpenRouter · Claude · DeepSeek · Tailscale · systemd · Telegram

Mark is a two-agent marketing system that runs autonomously on a dedicated Ubuntu server, handling the work that would otherwise need a full-time marketer across a portfolio of consumer apps. He wakes up, does research, writes copy, drafts content, and stops to ask for approval through Telegram before anything goes live.

The architecture is intentionally minimal. A working manager agent on a low-cost model (DeepSeek V4 Flash) handles orchestration, research, copywriting, and analytics. A separate strategist agent on a stronger model (Sonnet 4.6) produces the launch brief at the start and only reruns when there's real evidence the strategy isn't working. Everything else is a skill, not an agent. That cost-routing decision alone reduces token spend by roughly an order of magnitude versus running everything on a single frontier model.

I built the operational layer myself: agent identity files, marketing-research and copy-production skills, container orchestration with docker-compose, the Telegram approval gate, Tailscale for private networking, and systemd so the whole thing boots and recovers on its own.

Project S AI / Automation

Python · Gemma 3 27B · Llama 3.3 70B · Qwen3-TTS · Stable Diffusion XL · FFmpeg · OpenRouter

Project S is an end-to-end content generation pipeline that runs from a single command. One trigger kicks off scriptwriting, narration, imagery, and video assembly in sequence, and by morning a finished, fully-rendered video sits in the output folder.

The pipeline routes script generation across two LLMs through OpenRouter, synthesizes narration audio locally on my GPU using Qwen3-TTS, generates imagery through Stable Diffusion XL running in ComfyUI, and assembles the final video with FFmpeg using NVENC hardware encoding. The part that took the most work was the script quality loop: it evaluates drafts against a rubric and reruns until it hits a threshold, so the outputs don't feel templated. Most of the engineering ended up in places I didn't expect — VRAM contention between SDXL and the local LLM stage, FFmpeg's concat bug with AAC audio, banned-phrase detection that I eventually pulled out of the prompt and into Python because the model wouldn't reliably listen.

DealRadar — Agentic Restaurant Deal Finder AI / Full-Stack · 2026

FastAPI · React · Server-Sent Events · Google Places · DeepSeek V3 · OpenRouter

DealRadar finds restaurant deals near a zip code. The interesting part is how it decides where to look. The backend uses LLM tool calling so the model can choose between hitting Google Places for restaurant discovery, pulling from a local chain playbook I built, or kicking off a scrape and extraction pass for unknown restaurants. The chain playbook is a JSON database of pre-built deal extraction instructions for around eighty major chains. The FastAPI backend orchestrates the calls based on the model's choice and streams results back to the React frontend over SSE as they come in.

The hardest problem ended up being cost. Early on, every restaurant lookup hit the LLM, which killed the unit economics. The chain playbook system fixed that by skipping the model entirely for known restaurants — chains resolve in milliseconds, and only independents need an actual scrape and extraction pass. That one decision turned a fun demo into something that could actually run as a product.

Battery Retrofit Proposal Tool Internal Tool

Python · ReportLab · PyPDF2 · openpyxl · Excel / VBA

At my previous job, sales reps selling battery add-ons to existing solar customers had to submit a request to a back-office team and wait for someone to manually put together a proposal PDF. It could take hours, and by the time the proposal arrived the customer had often moved on.

I built a tool to fix that. The rep opens an Excel workbook, fills in five fields, picks the battery product and financing type, and clicks a button. Python reads the calculated pricing and loan outputs from the workbook via openpyxl, overlays all the customer information onto a branded PDF template using ReportLab, and writes a finished proposal to the output folder. The whole process takes a few seconds. Financial logic stays in Excel where the finance team can update it without touching any code.
